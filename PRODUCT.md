# Product

## Register

product

## Users

sub2api 站点管理员、风控运营和技术维护人员。他们通常在上线灰度、误杀排查、成本异常、上游模型故障或策略调整时使用这个工具，需要快速判断 Adapter 当前是否可靠、为什么某条输入被送审或阻断、以及配置变更会带来什么风险。

## Product Purpose

这个产品是 sub2api 风控中心和对话模型分类器之间的审计 Adapter。它通过 system prompt 模板、few-shot、`<user_input>` 包裹、JSON 分类、fail-open、决策缓存、分类映射和可观测指标，在控制成本和延迟的同时降低误杀风险。成功标准是：sub2api 可直接接入 `/v1/moderations`，管理员能在页面完成配置、测试、排障、审计和上线前检查。

## Brand Personality

克制、可信、清醒。界面语气要像生产运维工具：直接说明影响、风险和当前状态，不制造情绪，不用营销式承诺。

## Anti-references

不要像营销落地页、不要用大面积渐变或装饰图形、不要把复杂配置塞进一个 JSON 文本框、不要让危险开关看起来和普通设置一样、不要做成只有开发者才能读懂的裸 API 调试页。

## Design Principles

1. 先证明状态，再允许操作：概览、健康、Provider、fail-open、缓存和成本要能一眼判断。
2. 风险配置必须带上下文：高风险开关、抽样率、阻断分数和关键词修改要说明影响并写审计。
3. 调试链路要可复现：测试页面展示归一化文本、关键词、是否送审、Provider 摘要、映射分数和最终 JSON。
4. 默认保护敏感数据：页面不显示密钥明文，不默认展示完整用户输入，事件只显示 hash 和脱敏摘要。
5. 产品体验服务运维效率：密集但有秩序，控件和表格优先，避免装饰性布局。

## Accessibility & Inclusion

目标为 WCAG AA。正文对比度至少 4.5:1，状态不能只靠颜色表达，所有表单控件有明确 label，危险操作需要明确文本反馈，动效仅用于状态反馈并尊重 reduced motion。
